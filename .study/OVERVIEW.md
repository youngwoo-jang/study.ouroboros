# Ouroboros — 프로젝트 한눈 정리 (개념 OVERVIEW)

> 이 문서는 Ouroboros가 "무엇을, 왜, 어떻게" 해결하는지를 개념 수준에서 정리한 학습 노트다.
> 구현 디테일이나 코드 경로는 의도적으로 뺐다.

---

## 0. 한 줄 요약

Ouroboros는 **"명세 우선(Specification-First) AI 개발"** 워크플로우 엔진이다.
AI 코딩의 병목을 *모델 출력의 품질*이 아니라 **사람이 주는 입력의 모호성**이라고 진단하고,
"질문 → 명세 결정 → 실행 → 평가 → 진화"의 **자기참조 루프**(꼬리를 무는 뱀, 우로보로스)로
*코드를 짜기 전에 숨겨진 가정부터 드러내는* 시스템이다.

```
Interview → Seed → Execute → Evaluate
    ↑                           ↓
    └─── Evolutionary Loop ─────┘
```

---

## 1. 이 프로젝트가 해결하려는 문제

### 1.1 입력의 명확성 부재
"AI 코딩의 진짜 병목은 모델 능력이 아니라 사람의 명료함이다."
모호한 요구를 던지면 AI는 추측을 채워 넣고, 결과를 보고 다시 고치는 재작업 루프가 무한히 돌아간다.
"더 똑똑한 모델"이 아니라 "덜 모호한 입력"이 답이라는 가설에 베팅한 프로젝트다.

### 1.2 아키텍처 드리프트
명세 없이 곧장 구현으로 들어가면, 진행될수록 원래 의도에서 슬그머니 멀어진다.
PR 리뷰 단계에서 "어, 이거 우리가 원하던 게 아닌데?"가 드러나면 이미 비싸다.
구현 전에 의도를 **불변 명세(Seed)** 로 잠가버려 드리프트를 차단하는 게 해법.

### 1.3 검증의 비자동화
"보기엔 괜찮네"로 끝나는 수동 QA는 신뢰할 수 없다.
기계적 검사(테스트/빌드/린트)와 의미적 검사(요구사항을 정말 충족했나?)를 한 파이프라인으로 묶어 자동화하는 것이 목표.

### 1.4 정체와 순환
같은 가정을 반복하다 답이 제자리에 머물거나, A↔B를 왔다 갔다 하는 루프에 빠지는 문제.
이 패턴 자체를 감지해서 강제로 사고 방식을 점프시키는 장치가 따로 들어 있다.

### 한 줄로
> **AI 코딩은 출력이 아니라 입력에서 실패한다. Ouroboros는 코드 작성 전에 숨겨진 가정을 노출시키는 도구다.**

## 2. 어떻게 해결하는가 (개념적으로)

### 2.1 핵심 루프

```
Interview → Seed → Execute → Evaluate
    ↑                           ↓
    └─── Evolutionary Loop ─────┘
```

가장 중요한 차별점: **반복(repeat)이 아니라 진화(evolve)**.
평가 단계의 산출물이 다음 세대의 입력이 되어, N+1 세대는 N 세대보다 *덜 모호한* 명세를 갖는다.
같은 자리를 도는 게 아니라, 한 바퀴를 돌 때마다 한 단계 위로 올라가는 나선형 구조다.

### 2.2 단계별 역할

- **Interview (소크라테스식 질문)**
  사용자가 무엇을 만들고 싶은지를 직접 묻는 게 아니라, *그가 무엇을 가정하고 있는지*를 드러내는 질문을 던진다. "왜 그렇게 생각하시죠?", "그게 정말 같은 의미인가요?" 식. 모호도가 충분히 낮아질 때까지 계속한다.
- **Seed (불변 명세 결정화)**
  인터뷰 결과를 한 번에 결정화한 변경 불가능한 명세. 목표 / 제약 / 합격 기준 / 도메인 모델(온톨로지) / 종료 조건이 한 덩어리로 묶인다. 이후 단계는 모두 이 Seed를 기준으로 움직인다.
- **Execute (이중 다이아몬드 실행)**
  뒤에서 다시 설명. 핵심은 "발산-수렴"을 두 번 돌면서 문제 정의와 해법 설계를 분리하는 것.
- **Evaluate (3단계 검증 게이트)**
  값싼 검사부터 비싼 검사로 단계적으로 통과시키는 파이프라인.
- **Evolve (진화: Wonder → Reflect)**
  평가가 끝나면 "아직도 무엇을 모르는가?"를 묻고(Wonder), 도메인 모델을 어떻게 조정할지를 제안한다(Reflect). 그 결과가 다음 세대의 Seed가 된다.

### 2.3 두 개의 핵심 게이트 (시스템의 백본)

이 두 개의 임계값이 Ouroboros가 *언제 시작하고 언제 멈추는지*를 결정한다.

#### (a) 모호도 게이트 — 인터뷰는 언제 끝나는가
인터뷰 종료 조건은 "질문이 떨어졌을 때"가 아니라 **모호도가 충분히 낮아졌을 때**.
모호도는 목표의 명료함, 제약의 명료함, 성공 기준의 명료함을 가중평균한 값을 1에서 뺀 것.
기존 코드베이스 위에서 작업할 때(brownfield)는 "현재 컨텍스트를 얼마나 이해했는가" 차원이 추가로 들어간다.
이 임계값을 못 넘으면 Seed 생성 자체가 차단된다 → "아직 사람이 모호하게 말하고 있다"는 신호.

#### (b) 수렴 게이트 — 진화는 언제 끝나는가
진화 루프가 영원히 돌면 곤란하다. 종료 신호는 **도메인 모델의 안정화**.
세대를 거치면서 명세(목표·합격 기준)는 계속 다듬어지지만, 도메인 모델(이 시스템이 다루는 *무엇*) 자체는 점점 안정된 모양으로 수렴한다.
연속된 세대에서 도메인 모델이 거의 그대로 유지되면 → 시스템이 "자기가 무엇을 만들고 있는지" 알게 됐다는 뜻 → 자연스러운 종료점.
또한 같은 모양이 주기적으로 왔다 갔다 하면(순환) 별도로 감지하고, 너무 오래 진행되면 안전장치로 강제 종료.

> 직관: **명세는 진화하고, 온톨로지는 수렴한다.**

### 2.3.1 모호도 채점은 실제로 어떻게 이뤄지는가

모호도는 통째로 매기지 않고 **명료함을 여러 차원으로 쪼개 가중평균**한 뒤 1에서 뺀 값이다.

```
ambiguity = 1 - Σ(clarity_i × weight_i)
```

차원은 **목표(0.4) / 제약(0.3) / 성공기준(0.3)** 세 축이 기본이고, brownfield에서는 **컨텍스트(0.15)** 축이 추가된다(나머지는 0.35 / 0.25 / 0.25). 각 차원 점수는 LLM이 0~1로 판정하고, 점수와 함께 *근거 문장*을 같이 받아 추적 가능하게 둔다.

#### 실제 agentic flow

직관과 달리 **차원별로 subagent에 fan-out 하지 않는다**. 흐름은 두 역할이 직렬로 일하는 모양:

1. **Socratic Interviewer (질문자)** — 대화를 끌고 가며 질문만 던진다. 채점은 안 함.
2. **AmbiguityScorer (채점자)** — 사용자가 답할 때마다, 지금까지의 *전체 대화 트랜스크립트*를 통째로 LLM에 한 번 던지고, 모든 차원의 점수 + 근거를 **한 JSON으로 한꺼번에** 받는다.

```
유저 답변 → Scorer가 transcript 전체를 LLM에 1회 호출
       → {goal_clarity_score, goal_justification,
          constraint_clarity_score, constraint_justification, ...}
       → 가중평균 + 차원 하한 검사 → 게이트 통과/차단
```

차원을 쪼개 별도 호출로 보내지 않는 이유는 차원들이 서로 맥락을 공유하기 때문 — "목표"의 명료함은 "제약"이 무엇이냐에 따라 달라지므로, 한 호출 안에서 같이 보는 편이 일관성이 높다. 실패 시 재시도(JSON 파싱/토큰 잘림)는 최대 10회까지 같은 단일 호출 패턴을 반복한다.

게이트는 두 겹이다: **전체 모호도 ≤ 0.2**, 그리고 **각 차원이 자기 하한선**(목표 0.75, 제약 0.65, 성공기준 0.70 등)을 동시에 넘어야 한다. 한 차원이 너무 비어 있으면 평균이 통과해도 막힌다.

깔린 가정: ① 모호함은 몇 개 축으로 분해 가능하다, ② LLM이 자기 명료함을 판정할 수 있다, ③ 0.2는 경험치다. 한계: 모호도가 *낮다*는 게 *옳다*는 뜻은 아니다 — 분명히 틀린 명세도 0에 도달할 수 있고, 그건 평가 파이프라인이 따로 잡아야 한다.

### 2.4 이중 다이아몬드 (Double Diamond) 실행 모델

디자인 사고에서 차용한 패턴. 발산과 수렴을 두 번 반복한다.

```
Discover (발산)  →  Define (수렴, 온톨로지 필터)
   문제 공간 탐색      합의된 정의로 좁히기
        ↓                    ↓
Design (발산)    →  Deliver (수렴, 합격 기준 검증)
   해법 옵션 발산      구현 + AC로 좁히기
```

핵심 트릭: 발산 단계의 산출물을 **온톨로지(도메인 모델)** 라는 거름망으로 통과시킨다.
Seed가 정의한 "이 시스템이 다루는 개념들"에 부합하지 않는 아이디어는 걸러내고, 부합하는 것만 다음 단계로.
"무엇을 만드는가"를 명확히 정의해 두면, "어떻게 만드는가"의 발산이 통제 불능이 되지 않는다는 발상.

### 2.5 3단계 평가 파이프라인

비싼 검증을 항상 돌리지 않는 비용 최적화가 핵심.

1. **기계적(Mechanical) 검증** — 거의 공짜.
   린트, 빌드, 테스트, 커버리지, 정적 분석. 통과 못 하면 다음 단계로 가지 않는다.
2. **의미적(Semantic) 검증** — 보통 비용.
   합격 기준을 정말 충족했는가, 원래 목표와 얼마나 정렬돼 있는가, 의도에서 얼마나 멀어졌는가(드리프트)를 LLM이 판단.
3. **합의(Consensus) 검증** — 비싼 비용.
   여러 모델이 서로 다른 관점에서 투표. 의미적 검증이 애매하거나 사용자가 명시적으로 요청할 때만 발동.

### 2.6 9가지 에이전트 ("9 minds")

루프의 각 단계는 사실 서로 다른 *역할*의 사고가 필요하다. 한 모델이 모든 일을 하는 게 아니라, 역할별로 다른 페르소나를 부른다.

**핵심 5개**
- *소크라테스식 인터뷰어* — 질문만 한다. 절대 구현하지 않는다.
- *온톨로지스트* — "이게 정말 무엇인가?" 본질을 끄집어낸다.
- *Seed 아키텍트* — 인터뷰 결과를 불변 명세로 결정화한다.
- *평가자* — 3단계 검증을 집행한다.
- *QA 판정관* — 산출물 일반에 대한 합격/불합격 판정.

**측면 사고 5개 (정체 탈출용)**
- *반대론자(contrarian)* — "반대가 사실이라면?"
- *해커(hacker)* — "일단 돌아가게 만들고 우아함은 나중에."
- *단순화 전문가(simplifier)* — "가장 단순한 viable 버전은?"
- *연구자(researcher)* — "근거 있나? 데이터를 보자."
- *아키텍트(architect)* — "처음부터 다시 짠다면 이렇게 짤까?"

루프가 정체되거나 순환에 빠지면, 어떤 패턴인지에 따라 적절한 측면 사고 페르소나가 자동으로 호출되어 사고의 방향을 강제로 틀어버린다.

### 2.7 사용자 표면 — `ooo` 명령어

사용자는 사실 위 메커니즘을 직접 다루지 않고, 일련의 명령어로 호출한다 (개념만 추리면):

- `ooo interview` — 가정 노출 인터뷰
- `ooo seed` — 인터뷰 → 불변 명세
- `ooo run` — 명세를 받아 실제로 구현
- `ooo evaluate` — 3단계 검증
- `ooo evolve` — 한 세대만큼 진화
- `ooo ralph` — "통과할 때까지 멈추지 마"라는 끈질긴 루프
- `ooo unstuck` — 정체에 빠졌을 때 측면 사고 호출
- `ooo status` / `ooo drift` — 현재 상태와 의도로부터의 이탈 정도
- `ooo qa` — 범용 합격/불합격 판정
- `ooo pm` — PM 관점 인터뷰 → PRD 생성
- `ooo publish` — 명세를 GitHub 이슈/에픽으로 발행
- `ooo brownfield` — 기존 코드베이스 컨텍스트 자동 추출

각 명령어는 결국 위에서 설명한 단계 중 하나를 트리거하는 얇은 진입점이다.

### 2.8 시스템을 떠받치는 토대 (개념만)

- **비용 인지 라우팅** — 작업의 복잡도를 점수로 추정해서, 싼 모델 / 보통 모델 / 비싼 모델로 자동 라우팅. 실패하면 위로 올리고, 잘 풀리면 아래로 내린다.
- **비동기 작업 모델** — 오래 걸리는 작업을 호출자가 멍하니 기다리지 않게, 작업을 즉시 받아 ID를 돌려주고 변화가 있을 때만 알려주는 방식.
- **사건 소싱(Event Sourcing)** — 모든 상태 변화는 추가 전용 사건 로그로 남는다. 머신이 죽었다가 살아나도 진화 루프를 정확히 같은 지점에서 재개할 수 있다.
- **다중 런타임 추상화** — 같은 명세를 Claude Code, Codex, Hermes, OpenCode 같은 서로 다른 런타임에서 돌릴 수 있게 어댑터 계층으로 분리.
- **Brownfield 자동 탐지** — 프로젝트 매니페스트(언어/프레임워크 설정 파일)들을 훑어 기존 스택과 패턴을 자동으로 파악, 모호도 채점의 "컨텍스트" 차원으로 사용.

## 3. 데이터 흐름 한눈에 — 예시: "task 관리 CLI 만들기"

```
[사용자 목표: 모호함]
  "task 관리 CLI 만들어 줘"
        │
        ▼
[Interview]
  Q: 신규 프로젝트 / 기존 코드 위?     → 신규
  Q: 우선순위·상태 개념 필요?          → 둘 다 필요
  Q: 멀티유저인가?                     → 단일 사용자
  ...질문 반복...
  → 모호도 임계값 통과
        │
        ▼
[Seed]  (불변)
  목표:   단일 사용자용 Python CLI task 매니저
  제약:   외부 API 없음 / 로컬 SQLite
  AC:     생성 / 목록 / 완료 표시
  온톨로지: { Task: title, priority, status, created_at }
        │
        ▼
[Execute] (Double Diamond)
  Discover → Define(온톨로지로 거르기) → Design → Deliver
  → 코드 + 테스트 산출
        │
        ▼
[Evaluate]
  1단계 기계적: 린트 / 빌드 / 테스트 통과
  2단계 의미적: AC 100% 충족, 의도와 정렬됨
  3단계 합의: 생략 (충분히 확실)
  판정: 합격
        │
        ▼
[Evolve]  Gen 1 → Gen 2
  Wonder : "마감일 개념은?"
  Reflect: 온톨로지에 DueDate 추가 제안
  → 다음 세대 Seed 생성, 재실행, 재평가
        │
        ▼
[Convergence]  Gen 3
  도메인 모델이 직전 세대와 거의 동일
  → 안정화 신호, 루프 종료
```

## 4. 핵심 직관 정리 (외워두면 좋은 것)

- **출력이 아니라 입력을 고친다.** 모델이 아니라 명세가 병목.
- **명세는 진화하고, 온톨로지는 수렴한다.** 종료 신호는 후자에서 나온다.
- **반복이 아니라 진화.** 같은 자리를 도는 게 아니라 한 바퀴마다 한 칸 위로.
- **발산은 거름망이 있을 때만 안전하다.** 온톨로지가 그 거름망이다.
- **검증은 싼 것부터.** 비싼 합의 투표는 의미적 검증이 흔들릴 때만.
- **정체는 같은 사고로는 못 빠져나온다.** 그래서 측면 사고 페르소나가 따로 있다.
- **모든 사건은 기록된다.** 그래야 재시작 후 같은 지점에서 이어진다.

## 5. 다음 학습 단계 제안

이 OVERVIEW에서 출발해 `.study/`에 더 깊게 파볼 만한 토픽 (개념 학습 기준):

6. **정체 패턴 분류학** — 어떤 종류의 정체가 어떤 측면 사고로 풀리는가, 그 매핑은 자의적인가 필연적인가.
